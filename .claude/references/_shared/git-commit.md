# Conventional Commit Pattern

When a command commits work, use the project's commit-commands plugin (`/commit` skill) if available. Fallback below.

## Fallback (no /commit skill)

1. `git status` to confirm staged set.
2. Generate a conventional commit message from the diff:
   - `feat:` new features
   - `fix:` bug fixes
   - `refactor:` code improvements
   - `docs:` documentation
   - `test:` test additions/changes
   - `chore:` maintenance / framework updates

3. Pass the message via HEREDOC:

   ```bash
   git commit -m "$(cat <<'EOF'
   <type>: <subject>

   <optional body>
   EOF
   )"
   ```

## Hard rules

- Never commit unless the user explicitly asked or the command spec mandates it (e.g., `/ship` Step 2, `/evolve` Step 7).
- Never `--amend` an already-pushed commit unless the user requests it.
- Never `--no-verify` unless the user requests it.
- Never include `.env`, credentials, or large binaries in a commit.
- Stage files by name; avoid `git add -A` / `git add .` unless the working tree was just generated by the command itself.
