# v0.6.3 — Three-way `/merge-configs` with mandatory post-merge budget guard

## Why

v0.6.0 codified line-count budgets and v0.6.2 made the lint reachable from every consumer project, but `/merge-configs` itself was never required to honor them. The strategy table said things like "Append user additions; dedupe" — naive append. A user with a pre-v0.5 inline rule of 100 lines merging into a slim 30-line framework rule produced a 130-line file: budget-blowing, while the framework's own rules said "stay under 60." The cap was theatre.

This release plugs that hole.

## Added — three-side merge model

- **`merge-strategy.md` § Three sides of a merge** makes the user / framework / base distinction explicit. The base is reconstructed via `git show v<previousVersion>:<path>` against this repo when `.claude/.init-meta.json#previousVersion` is known; absent that, the merger falls back to a two-way user-vs-framework merge. The base lets the merger tell "user customised this" from "framework changed this" — which decides whether a difference is a user edit to preserve or a framework edit to take.

## Added — mandatory post-merge budget guard

- **`merge-strategy.md` § Post-merge budget guard** — every category-specific merge operation MUST run `node cli/file-size-check.js --json` on its output. The decision matrix:
  - `level=ok` → continue.
  - `level=warn` → continue, surface as `size-warn=N` in the final one-liner.
  - `level=block` → blocker; route the merged file through `merge-configs/05-decompose.md`. Decomposition is constrained to relocate, never delete — both user and framework content survive.
  - Cannot resolve in one decomposition pass → manual-resolution blocker; `.backup` stays in place so the user can re-attempt.

Settings (`.claude/settings.local.json`) is the only category exempt — `.json` is out of scope for the line-count lint.

## Changed — bullet-level rule merges

`merge-configs/04-execute.md` Rules section now spells out the three-way bullet algorithm:
1. Union of bullets across user and framework, deduped by normalized text.
2. Framework-edit-since-base bullets → take framework. User-edits-since-base → take user. Both-changed-the-same-bullet → diff prompt.
3. Write the merged rule.
4. **Post-merge budget guard runs.** Overflow goes to `.claude/references/<domain>-detail.md` under a dated `## User-merged: YYYY-MM-DD` heading; cited from the rule's `## References`. Bullets are relocated, never dropped.

## Changed — decomposition routine accepts post-merge files

`merge-configs/05-decompose.md` now documents two trigger modes:
1. **Pre-merge** (Step 3 of `/merge-configs`): user-side file already over budget, or pre-v0.5 install.
2. **Post-merge** (Step 4 budget guard): a category merge produced a result the lint blocked.

The routine itself is identical in both modes — only the input source differs.

## Changed — cleanup gated on the budget guard

`/merge-configs` Step 6 now refuses to delete a `.backup` file unless the corresponding merged file passed the budget guard (lint exit 0 or 1). Files still over the hard cap after one decomposition pass are surfaced as manual-resolution blockers; `.claude/.init-meta.json` is only deleted when every backup cleared.

## Changed — audit log

When any decomposition runs (pre- or post-merge), `/merge-configs` appends to `.claude/.last-merge-log.md` listing each section moved: source path:line-range → destination path:line-range, side: `user` or `framework`. The user can review and revert from `.backup` files until Step 6 cleans them up.

## Migration

```bash
npx ai-development-framework update
```

Then re-run `/merge-configs` if you have any `.backup` files in your project. The new flow is strictly additive — files that were already lean still take the splice path; only over-budget merges trigger decomposition.
